### API

## A — windows-client（C#）接入 Phase 0 ✅ 已完成

### 已交付（`apps/windows-client`）
- **`CadModels.cs`**：`CadManifest`/`CadArtifact`/`CadMcpServer`/`CadBundleEntry`/`CadLocalState` DTO。
- **`CadMcpConfigWriter.cs`**：幂等写 `# BEGIN/END CODEX CAD MCP` 块，照 `CodexConfigWriter` 范式，TOML 单引号串处理 Windows 路径，与 gateway 块并存。
- **`CadSync.cs`**：拉 manifest → 比对 `revision` → 逐工件**双重 sha256 校验**（内容 + `x-artifact-sha256` 响应头）→ 增量下载 → zip 解压落地（skills→`~/.codex/skills/<id>/`、scripts/bundles→本地目录）→ 解析 `bundleId`（读 bundle 内 `mcp-entry.json`）为本机 command/args → 写 config。本地状态存 `%AppData%/CodexLoginTools/cad/state.json`，支持增量与清理已删工件。
- **`MainForm.cs`**：`ToggleGatewayAsync` 启动代理后调 `SyncCadCapabilitiesAsync()`（失败仅提示、不阻断代理）；停止时 `CadSync.RemoveMcpConfig()` 移除受管块。
- **`.csproj`**：版本 `0.2.2 → 0.2.3`。

### 关键设计决策
- **stdio MCP 进程由 Codex 依 config.toml 自行拉起**，客户端只"下载+校验+落地+写配置"，不自己 spawn，简化实现且符合 Codex MCP 机制。
- **CAD 同步失败不阻断代理启动**——属可选增强能力。
- **bundle 缺 `mcp-entry.json` 时跳过该 server**，不写半截配置。

### 验证证据
- `dotnet build -c Release`：**成功，0 警告 0 错误**（net9.0-windows，2026-07-09）。

### 未验证（需真机）
- 真机端到端：拉包→config 写入→Codex 启动本地 MCP→AutoCAD 出图。依赖 Phase 2/3 产出的 bundle 与 AutoCAD 环境。

## 待办与交接

### bundle 打包约定（供 Phase 2/3 产出 MCP 包时遵守）
- 每个 bundle 为 zip，解包后根目录需含 `mcp-entry.json`：
  `{ "command": "python", "args": ["-m","cad_mcp.server"], "startupTimeoutSec": 30 }`
- `command` 为相对路径时按 bundle 解包目录解析为绝对路径（用于自包含 exe）。
